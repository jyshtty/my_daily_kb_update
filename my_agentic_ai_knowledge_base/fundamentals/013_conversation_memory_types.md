# ConversationBufferWindowMemory & ConversationSummaryMemory

Both solve the same problem — keeping conversation history within the LLM's context window — but with different strategies.

## ConversationBufferWindowMemory

Keeps only the last `k` conversation turns. Older turns are dropped entirely.

```python
from langchain.memory import ConversationBufferWindowMemory
from langchain.chains import ConversationChain
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

memory = ConversationBufferWindowMemory(k=3)  # keep last 3 turns

chain = ConversationChain(llm=llm, memory=memory, verbose=True)

chain.predict(input="Hi, my name is Ajay.")
chain.predict(input="What is LangChain?")
chain.predict(input="What is LCEL?")
chain.predict(input="What is my name?")  # still knows — within last 3 turns
```

### Key parameters

| Parameter      | Default | Description                                  |
|---------------|---------|----------------------------------------------|
| `k`           | 5       | Number of recent turns (human + AI = 1 turn) |
| `return_messages` | False | Return as message list instead of string  |
| `memory_key`  | `"history"` | Key injected into the prompt              |

### When to use

Simple chatbots where recent context is enough and you need a hard cap on tokens.

---

## ConversationSummaryMemory

Instead of dropping old turns, it uses the LLM to **summarise** past conversation and keeps growing the summary as new turns arrive.

```python
from langchain.memory import ConversationSummaryMemory
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

memory = ConversationSummaryMemory(llm=llm)

chain = ConversationChain(llm=llm, memory=memory, verbose=True)

chain.predict(input="Hi, I'm building a RAG pipeline.")
chain.predict(input="I'm using FAISS as the vector store.")
chain.predict(input="What have we discussed so far?")
# LLM summarises: "User is building a RAG pipeline using FAISS."
```

### Key parameters

| Parameter      | Default | Description                                       |
|---------------|---------|---------------------------------------------------|
| `llm`         | required | LLM used to generate summaries                   |
| `memory_key`  | `"history"` | Key injected into the prompt                 |
| `max_token_limit` | None | Optionally cap summary token length           |

### When to use

Long conversations where you can't afford to drop context — the summary retains the *meaning* of past turns without storing raw text.

---

## Side-by-side comparison

| Feature                     | BufferWindowMemory       | SummaryMemory                  |
|-----------------------------|--------------------------|--------------------------------|
| How history is stored       | Raw message text         | LLM-generated summary          |
| Token usage                 | Fixed (bounded by `k`)   | Grows slowly with conversation |
| Information loss            | Hard — old turns dropped | Soft — summary may lose detail |
| Extra LLM calls             | No                       | Yes (one per summary update)   |
| Best for                    | Short-context tasks       | Long-running conversations     |

## Common interview questions

**Q: What is the difference between ConversationBufferMemory and ConversationBufferWindowMemory?**  
`BufferMemory` stores the entire history (unbounded). `BufferWindowMemory` keeps only the last `k` turns, discarding older ones.

**Q: Why does ConversationSummaryMemory need an LLM passed in?**  
It needs to call the LLM to compress past conversation into a running summary each time a new turn is added.

**Q: What is the risk of ConversationSummaryMemory?**  
The summary is generated by the LLM, which may lose specific details (names, numbers, exact wording). It preserves intent but not verbatim content.

**Q: Which memory type would you use for a customer support bot with very long sessions?**  
`ConversationSummaryMemory` — it avoids context overflow while retaining the gist of the conversation. For critical details, combine with `ConversationSummaryBufferMemory`, which keeps recent turns verbatim and summarises older ones.
